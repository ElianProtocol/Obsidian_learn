Это наборы правил: кто, куда и когда сливает код.


1.. ==*Trunk-Based Development*==

   Самый быстрый способ. Разработчики пушат мелкие правки напрямую в основную ветку (main / master) или используют очень короткоживущие ветки ( на 1-2 часа) 
   
		Плюсы: Нет ада со слиянием (merge hell), высокая скорость
		
		Для кого: Опытные команды с мощным автоматическим тестированием.

![[Pasted image 20260430012027.png|606]]

2.. ==*GitHub Flow*==

Простая и популярная стратегия.

	1. Создаем ветку под задачу ( feature-branch ).
	2. Пишем код, пушим.
	3. Создаем Pull Requests ( запрос на вливание ).
	4. После код-ревью и тестов вливаем в main.
	Для кого: Облачные сервисы, где деплой происходит постоянно

![[Pasted image 20260430012426.png]]

3.. ==*GitFlow*==

Классическая, но "тяжелая" стратегия. Есть две вечные ветки:

	master: только готовые релизы
	develop: здесь идет основная разработка
	Дополнительно: feature- , release- и hotfix- ветки.
	Для кого: Крупные энтерпрайз-проекты с четким графиком релизов
