# Основные методологии: Scrum, Kanban, XP

В DevOps-среде методология определяет, как поток работы движется от идеи до эксплуатации. Выбор между Scrum, Kanban и Extreme Programming (XP) влияет на предсказуемость поставки, качество кода, скорость обратной связи и способы реагирования на инциденты.  

## Scrum

Scrum — итеративно-инкрементный подход с фиксированной длительностью спринтов. Ключевые элементы:  

- Роли: команда разработки, владелец продукта, скрам-мастер.
- События: планирование спринта, ежедневный митинг, обзор, ретроспектива.
- Артефакты: бэклог продукта, бэклог спринта, инкремент.

Особенности для DevOps:  

- Инфраструктурные и операционные задачи включаются в бэклог на равных с фичами.
- Непредвидимая работа (инциденты) обрабатывается через резерв емкости спринта или отдельный поток.
- Definition of Done расширяется до “готово в продакшене, наблюдаемо и алертируемо”.
- Метрики: скорость (velocity) для предсказуемости, процент невыполненных задач, доля технического долга.

Когда полезен: стабильный входящий поток, фокус на планируемых инкрементах, синхронная работа кросс-функциональной команды.  

## Kanban

Kanban — система управления потоком, основанная на ограничении незавершённой работы (WIP) и визуализации стадий.  

- Поток непрерывный, без фиксированных итераций.
- Политики явны: критерии входа/выхода, WIP-лимиты, классы обслуживания.
- Улучшение процесса через анализ узких мест.

Особенности для DevOps:  

- Естественно совместим с обработкой инцидентов и запросов на обслуживание.
- Метрики: цикл (cycle time), время поставки (lead time), WIP, пропускная способность (throughput), стабильность потока (flow efficiency).
- Экспедит-класс для критических инцидентов без нарушения общих лимитов.

Когда полезен: высокий уровень непредсказуемости, постоянный приток задач, акцент на времени прохождения.  

## Extreme Programming (XP)

XP — набор инженерных практик, повышающих качество и скорость обратной связи:  

- Непрерывная интеграция, тестирование первым (TDD), парное программирование, рефакторинг.
- Простота дизайна, коллективное владение кодом, автоматизация сборки и деплоя.
- Частые малые релизы и работа по trunk-based подходу.

Особенности для DevOps:  

- Улучшает устойчивость CI/CD-пайплайнов, снижая дефекты на ранних стадиях.
- Поддерживает фича-флаги и канареечные выкладки.
- Требует дисциплины и высокого уровня автоматизации.

Когда полезен: строгие требования к качеству, высокие риски регрессий, быстрые итерации с малым батч-сайзом.  

## Сравнение

|Критерий|Scrum|Kanban|XP|
|---|---|---|---|
|Ритм|Фиксированные спринты|Непрерывный поток|Непрерывный, частые релизы|
|Планирование|По спринтам|По мере готовности|Непрерывное, мелкие изменения|
|Управление нагрузкой|Коммитмент на спринт|WIP-лимиты|Мелкие батчи, практики качества|
|Работа с инцидентами|Резерв/отдельный поток|Класс обслуживания “экспедит”|Малые изменения + быстрая проверка|
|Основные метрики|Velocity, прогноз|Lead/Cycle time, WIP, throughput|Покрытие тестами, частота интеграций|
|Фокус|Предсказуемость инкрементов|Стабильный поток и время прохождения|Качество и обратная связь|

## Пример явных политик (Scrumban-подход)

```yaml

workflow:
  columns:
    - name: Backlog
      wip: 999
      entry: описано, оценено
    - name: In Progress
      wip: 5
      entry: готовы тесты, фича-флаг создан
    - name: Code Review
      wip: 3
      exit: 2 approvals, зелёный CI
    - name: Testing
      wip: 3
      exit: автотесты пройдены, мониторинг настроен
    - name: Deploy
      wip: 2
      policy: canary 10% → 50% → 100%
definition_of_done:
  - деплой автоматизирован
  - алерты по ключевым SLI настроены
  - откат скриптом возможен
cadence:
  planning: еженедельно 30 минут для приоритезации
  retro: каждые 2 недели, улучшения в policies
classes_of_service:
  - standard (SLA 5 дней)
  - expedite (SLA 24 часа, WIP не учитывается, до 1 одновременно)
```
