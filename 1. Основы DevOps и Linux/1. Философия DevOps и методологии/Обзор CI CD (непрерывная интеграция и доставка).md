# Обзор CI/CD: непрерывная интеграция и доставка

Непрерывная интеграция и доставка — это подход к разработке и эксплуатации, при котором путь от изменения в коде до его безопасного развертывания автоматизируется. Цель — уменьшить время поставки, снизить количество ошибок и обеспечить предсказуемость релизов через короткие итерации и постоянную обратную связь.  

## Термины и границы понятий

- CI (Continuous Integration) — автоматическая сборка и проверка каждого изменения, попадающего в репозиторий. Основной акцент на раннем обнаружении дефектов.
- Continuous Delivery — автоматизированная поставка готовых к релизу артефактов до предпроизводственных сред с контролируемым, часто ручным, переводом в продакшен.
- Continuous Deployment — автоматический выпуск в продакшен при прохождении всех проверок без ручного шага одобрения.

Сравнение:  

|Аспект|CI|Continuous Delivery|Continuous Deployment|
|---|---|---|---|
|Триггер|Каждый коммит/merge|Сборка релизного артефакта|Успешный пайплайн|
|Цель|Проверка качества|Готовность к выпуску|Немедленный выпуск|
|Контроль|Автотесты, линтеры|Артефакты, инфраструктура|Политики и метрики в прод|
|Ручные действия|Нет|Возможны approvals|Не требуются|

## Цели и измерения

- Сокращение lead time от коммита до развертывания.
- Увеличение частоты релизов при сохранении стабильности.
- Снижение change failure rate благодаря автоматизированным проверкам.
- Быстрое восстановление (MTTR) за счет воспроизводимых релизов и откатов.

## Элементы конвейера

- Триггеры: push, merge request, расписание, тег релиза.
- Исполнители (runners/agents): изолированные среды выполнения шагов.
- Этапы: build, test, security scan, package, release, deploy.
- Артефакты: собранные пакеты/образы, сохраняемые в репозиториях (artifact/storage/registry).
- Среды: dev → stage → prod, с продвижением (promotion) из одного набора артефактов.
- Политики: quality gates, approvals, правила веток, условия для деплоя.
- Кэширование: ускорение сборок за счет reuse зависимостей.
- Управление секретами: внешние хранилища и инъекция в рантайме.

## Типичный поток

1. Коммит в основную ветку или merge request.
2. Сборка и линтинг; статический анализ.
3. Автотесты: unit, интеграционные, контрактные.
4. Сканирование безопасности и лицензий.
5. Пакетирование и публикация артефактов.
6. Развертывание в stage с прогонкой smoke/acceptance-тестов.
7. Одобрение или автоматическое развертывание в prod по политике.

## Пример декларативного пайплайна

```yaml

stages:
  - build
  - test
  - package
  - deploy

variables:
  APP_VERSION: "${CI_COMMIT_SHORT_SHA}"

build:
  stage: build
  script:
    - make deps
    - make build
  artifacts:
    paths:
      - dist/
    expire_in: 1 week

test:
  stage: test
  needs: ["build"]
  script:
    - make lint
    - make unit
    - make integration
  cache:
    paths:
      - .cache/pip

package:
  stage: package
  needs: ["test"]
  script:
    - docker build -t registry.example/app:$APP_VERSION .
    - docker push registry.example/app:$APP_VERSION
  artifacts:
    reports:
      dotenv: release.env

deploy_stage:
  stage: deploy
  needs: ["package"]
  script:
    - ./scripts/deploy.sh stage $APP_VERSION
  environment: "stage"

deploy_prod:
  stage: deploy
  needs: ["deploy_stage"]
  when: manual
  only:
    - main
  script:
    - ./scripts/deploy.sh prod $APP_VERSION
  environment: "production"
```

## Паттерны и организационные решения

- Trunk-Based Development с короткими ветками и частыми интеграциями.
- Feature flags для безопасного включения функций без долгих релизных веток.
- Иммутабельные артефакты: один и тот же образ/пакет продвигается между средами.
- Политики защиты веток и обязательные статусы проверок до слияния.

## Типичные анти-паттерны

- Долгоживущие ветки и редкие слияния, приводящие к конфликтам и регрессиям.
- «Хрупкие» тесты и нестабильные окружения раннеров.
- Вшивание секретов в репозиторий и артефакты.
- Ручные, неконтролируемые шаги между сборкой и развертыванием.
- Пересборка артефактов для разных сред вместо продвижения одного артефакта.
